Mapa de Arquitetura · Desktop → Web → Mobile

Orkestral hoje é um backend Node empacotado num Electron. Pra virar web e mobile, o trabalho é separar esse backend do app.

O processo main do Electron já É um servidor completo (DB, execução de agentes, MCP, schedulers). O renderer é a UI. O caminho pra web/mobile é promover o main a um servidor hospedado e trocar a ponte IPC por HTTP/WebSocket. Boa notícia: vários pedaços já são remotos.

⚠ O acoplamento é o maior trabalho da UI

window.orkestral[...] é chamado cru, inline, em 105 arquivos / ~592 lugares, 304 canais distintos. Não há camada de abstração. Primeiro refactor obrigatório: extrair uma interface OrkestralApiClient (1 interface, 3 implementações: IPC, HTTP, mobile) e um OrkestralEventBus pros eventos. Sem isso, web e mobile não compartilham nada.

03O bloqueador central

Como os agentes executam — e por que isso não roda no navegador/celular

Todo adapter principal é literalmente um child_process.spawn("claude" | "codex" | "cursor-agent"): binários que precisam existir no PATH da máquina, com acesso ao repositório em disco. Não existe equivalente REST desses CLIs.

Adapter Como roda Onde pode viver
claude_local spawn("claude" …stream-json) host com CLI
codex_local spawn("codex" exec --json) host com CLI
cursor_local spawn("cursor-agent") host com CLI
orkestral_local (Forge) node-llama-cpp in-process, GGUF ~4.7GB máquina/GPU
openclaw_gateway WebSocket RPC p/ host remoto (ED25519) já é remoto
cursor_cloud SDK remoto (stub hoje) remoto qdo pronto
★ O padrão pra web/mobile já existe no código: openclaw_gateway

É um protocolo WebSocket completo (v3, auth por device, frame-RPC) que manda o prompt pra um host remoto e faz streaming da saída de volta — exatamente o modelo "Orkestral como cliente, execução como serviço remoto". O destino natural é um runner host por workspace: uma VM/container com os CLIs + git + repo + o backend Orkestral rodando, que o app web/mobile acessa por WebSocket recebendo os mesmos eventos chat:stream / issue:execution-event / terminal:data.

5 travas absolutas de portabilidade

  • Spawn dos CLIs (claude/codex/cursor) — precisam estar instalados no host
  • Filesystem do repo (CWD) — agente lê/escreve arquivos, roda git/tsc/php -l
  • node-pty — terminal real, exige PTY do SO (impossível no browser)
  • node-llama-cpp / GGUF — binário nativo Metal/CUDA, modelos GB
  • Servidor MCP interno em localhost — dá as ferramentas ao CLI; tem que estar no mesmo host do agente

Schedulers de fundo (precisam de processo vivo)

  • Heartbeat (60s) — delega issues / roda resumo Forge por agente
  • Routine (60s) — dispara prompt agendado por rotina
  • Monitor (60s) — re-executa issues por cron
  • Forge update (24h) + flush de signals — telemetria pro cloud

No modelo hospedado viram jobs do servidor/runner — não somem.

04Camada de dados

Já é multi-workspace; falta ser multi-usuário remoto

Estado atual

SQLite (better-sqlite3, síncrono) + Drizzle, arquivo único em ~/.orkestral/…/orkestral.db. 73 migrations manuais por user_version. ~60 tabelas em 7 domínios (agents, issues, chat, kb, integrações, observabilidade, identidade).

Bom: workspaceId é a coluna de tenancy em quase tudo, com ON DELETE CASCADE. A estrutura já é escopada por workspace.

10 lacunas pra backend remoto

  • crítico sem user_id / sem contas de usuário
  • crítico sem auth/sessão (cloud account é blob JSON em settings)
  • crítico tokens cifrados com safeStorage (keychain do SO) — não migra direto
  • crítico SQLite síncrono → portar pra Postgres
  • alto caminhos de filesystem salvos no DB (workspaces.path, sources.path…)
  • alto embeddings = GGUF local + cosine em memória (cap 5000)
  • alto auth do WhatsApp em arquivos do Electron, não no DB
  • médio sem camada de org acima do workspace (billing/time)

05MCP, Skills e Integrações

MCP remoto/URL já é suportado — não é só plano

O servidor MCP do Orkestral já é HTTP (roda em 127.0.0.1:porta com token), não stdio. E buildMcpConfigForRun() em chat-service.ts já emite os dois transportes no mesmo formato:

REMOTO (o futuro)

{ type:'http', url, headers }

Transporte McpTransport = 'stdio' | 'http' | 'sse' é tipo de produção. A infra de MCP por URL está operacional — falta os catálogos apontarem pra ela. É exatamente a direção "MCP abre a URL dele, não no app".

LOCAL (hoje, maioria)

{ command, args, env }

Os 22 MCPs do catálogo curado ainda são stdio (processos npx). Esses precisam de host; os HTTP/URL portam limpo pra web/mobile.

Integrações — a maioria já é API remota

Remoto (porta limpo) Local (precisa de host)
GitHub, Sentry, Azure DevOps, New Relic, Better Stack, Telegram (Bot API), Cloud Auth (Supabase, já tem web em orkestral.pro), Forge Signals, OpenClaw Gateway, PulseMCP WhatsApp (baileys, WebSocket local), Voz (GGUF local), Docker, Terminal, Screenshot, embeddings locais, secrets via safeStorage

Catálogo da marketplace: 3 tiers — bundled estático (offline), PulseMCP (16k+ MCPs, com version-pinning anti-supply-chain) e GitHub skills. Já existe hook de env pra um registry hospedado futuro (ORKESTRAL_MARKETPLACE_*_API).

06App mobile — escopo realista

Celular é "comando e acompanhamento", não IDE

Mobile não vai rodar terminal, IDE, docker ou git local — isso fica no desktop/web. O valor no celular é: acompanhar agentes, aprovar/revisar, conversar, ver observabilidade e disparar trabalho que executa no runner remoto.

Entra fácil

  • Home / Dashboard
  • Chat / Session
  • Inbox (propostas)
  • Issues + detalhe
  • Goals
  • Agents (lista/config)
  • Skills / atribuição
  • Costs · Activity
  • Sentry / Observability
  • Channels (status WhatsApp/Telegram)

Dá, com trabalho

  • Code reviews (ler + aprovar)
  • Issue detail com diff (read-only)
  • Integrations / Providers (OAuth mobile)
  • Push notifications de eventos

Fica no desktop

  • Sources / IDE (CodeMirror)
  • Terminal (xterm/pty)
  • Docker
  • Logs brutos
  • Knowledge graph editor
  • Code changes (edição)
Stack mobile recomendada

React Native + (NativeWind p/ reaproveitar Tailwind) consumindo o mesmo OrkestralApiClient e OrkestralEventBus via HTTP/WebSocket. Reaproveita lógica (tipos de src/shared, hooks de dados, i18n, stores não-desktop) — re-faz a árvore de UI (componentes nativos). Navegação: React Navigation.
